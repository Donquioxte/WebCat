SQL Injection to Domain Admin — A Full Internal Network Pwn

Starting from a blind SQL injection in a login form, this HackTheBox Pro Lab walkthrough covers the full path from initial foothold to domain administrator.

Published February 20, 2026 Reading time 18 min Author Wil

From a login form to the entire domain

This Pro Lab is a good example of why a single web application vulnerability should never be assessed in isolation. What started as a blind SQL injection in an internal HR portal's login form eventually led to full domain administrator access across the lab's Active Directory environment. No individual step in this chain was especially exotic — the value of the write-up is in the path itself, and what it says about how real internal compromises tend to unfold.

Finding the SQL injection

The HR portal's login form was the obvious starting point. Submitting a single quote in the username field returned a generic error page rather than a database error — meaning this wasn't going to be a simple error-based injection. Time-based blind testing confirmed the parameter was injectable:

POST /login HTTP/1.1 Host: hr.corp.local username=admin' AND SLEEP(5)-- -&password=x

A five-second delay in the response confirmed the injection point. From here, sqlmap handled extraction far more efficiently than manual time-based queries:

sqlmap -u "http://hr.corp.local/login" \ --data="username=admin&password=x" \ -p username --technique=T --batch --dump

The dumped credentials table contained hashed passwords for several internal accounts, including one belonging to an IT support user. The hash cracked quickly against a standard wordlist — a reminder that even when the injection itself is sophisticated, the resulting credentials are often only as strong as the organisation's password policy.

Establishing a foothold

The cracked IT support credentials worked for both the HR portal and, after testing, for SMB authentication on the internal network — credential reuse across systems that should have been segmented. Enumerating accessible shares with these credentials surfaced a writable directory used for IT support ticket attachments.

# Enumerate accessible SMB shares with the recovered credentials smbclient -L //10.10.20.5 -U itsupport%CrackedPassword123 # Writable share found — used for ticket attachments smb: \> put shell.aspx

The ticketing web application, accessible on the internal network, would periodically process new attachments in that directory — and critically, files with an .aspx extension placed there were directly web-accessible and would execute. Uploading a simple ASP.NET web shell and browsing to it provided code execution on the ticketing server.

Why this kind of gap is common internally Internal applications are frequently held to a lower security bar than public-facing ones, on the assumption that the network perimeter is the real defence. File upload validation, in particular, is often weaker on internal-only tools — exactly the kind of gap that becomes critical once an attacker has any foothold at all on the internal network.

Lateral movement and credential harvesting

With code execution on the ticketing server, the next step was harvesting additional credentials from memory. The server had been joined to the domain, and a domain service account had recently authenticated to it for a scheduled task — leaving recoverable credential material in memory.

# Dumping credentials from LSASS memory using Mimikatz mimikatz # privilege::debug mimikatz # sekurlsa::logonpasswords

This surfaced NTLM hashes for a domain service account — svc_backup — used for scheduled backup jobs across multiple servers. A quick BloodHound collection against the domain revealed that this account, while unprivileged on its face, had been granted GenericAll rights over a security group that was nested inside Domain Admins — almost certainly an oversight from a past permissions change that was never cleaned up.

# Using the harvested hash to authenticate without cracking it evil-winrm -i 10.10.20.10 -u svc_backup -H NTLM_HASH_HERE

From a nested group to Domain Admin

The GenericAll permission on the security group meant svc_backup could add arbitrary members to that group — including itself. Since the group was nested inside Domain Admins, adding svc_backup to it was functionally equivalent to granting it domain administrator privileges directly.

# Abusing GenericAll to add svc_backup to the privileged nested group net rpc group addmem "Backup Operators Delegated" svc_backup \ -U svc_backup -S 10.10.20.10

BloodHound's attack path graph made this privilege escalation route obvious almost immediately once the collection data was loaded — this is exactly the kind of indirect, multi-hop permission relationship that is extremely difficult to spot by manually reviewing group memberships, but trivial to identify once visualised as a graph.

Tip from practice Run a BloodHound collection as early as possible once you have any valid domain credentials, even low-privileged ones. Nested group permissions like this one are one of the most common privilege escalation paths in real Active Directory environments, and they're consistently underestimated during manual review.

From there, authenticating as svc_backup with effective Domain Admin rights provided full control over the domain — DCSync rights to extract every credential hash in Active Directory, and the ability to authenticate to any domain-joined machine, including the domain controller itself.

# Extracting the krbtgt hash and all domain credentials via DCSync secretsdump.py corp.local/svc_backup@10.10.20.1 -hashes :NTLM_HASH_HERE

Why this chain matters

Every step in this path, taken individually, is a known and well-documented technique. What makes the overall chain dangerous is how naturally it flows from one weakness to the next: weak credential hygiene led to internal access, lax file upload controls on an internal tool led to code execution, normal administrative activity left credentials recoverable in memory, and a forgotten nested group permission turned an ordinary service account into a path to full domain compromise.

This is the pattern behind most real-world Active Directory compromises. Attackers rarely need a single dramatic zero-day — they need a handful of individually unremarkable misconfigurations that happen to connect.

How this gets fixed

  • Parameterised queries everywhere — the SQL injection at the start of this chain should never have been possible with modern query builders or ORMs in use.
  • Unique credentials per system, enforced through a password manager or SSO, so a single cracked hash doesn't grant access across unrelated services.
  • Strict file upload validation on internal applications held to the same standard as public-facing ones — extension allow-lists, content-type verification, and storing uploads outside any web-executable directory.
  • Credential Guard or equivalent LSASS protections to make in-memory credential harvesting significantly harder even with local administrative access.
  • Regular BloodHound audits of the production Active Directory environment to catch exactly this kind of unintended nested-group privilege escalation path before an attacker does.

Takeaway

This lab is a good reminder that internal network security deserves the same rigor as the public perimeter — and that Active Directory privilege escalation paths are rarely a single obvious misconfiguration. They're usually a chain of small, individually defensible decisions that nobody connected until it was too late. If you're learning Active Directory attack paths, getting comfortable with BloodHound early will save you far more time than trying to manually trace permission relationships by hand.
