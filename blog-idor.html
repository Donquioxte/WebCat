Why IDOR Vulnerabilities Are Still Everywhere in 2026

Insecure Direct Object References have been on the OWASP Top 10 for years. So why do they keep showing up in modern applications — including ones built by teams that know better?

Published May 28, 2026 Reading time 8 min Author Wil

The vulnerability that refuses to die

There is something almost embarrassing about IDOR. It is not a complex memory corruption bug or a cryptographic weakness requiring a PhD to understand. At its core, it comes down to a single question the application forgot to ask: does this user actually own this resource?

And yet, year after year, IDOR consistently ranks among the most commonly reported vulnerabilities in bug bounty programs and real-world penetration tests. In 2026, with AI-assisted development, massive API ecosystems, and teams that have access to more security tooling than ever before, the problem has not shrunk. If anything, it has more surface area to hide in.

This post breaks down what IDOR is, why it keeps happening, and what secure authorization actually looks like in practice.

What is IDOR?

IDOR stands for Insecure Direct Object Reference. It occurs when an application exposes a reference to an internal object — a database record, a file, a user account — and fails to verify whether the requesting user has permission to access it.

The classic example looks like this. You log into a web application and notice your profile URL:

GET /api/users/1042/profile Authorization: Bearer eyJhbGciOiJIUzI1NiJ9...

You change 1042 to 1043 and send the request. If the server returns someone else's profile data without any error, that is an IDOR vulnerability. The application authenticated you correctly — it knows who you are — but it never checked whether you were allowed to view user 1043's data.

Authentication vs. Authorization These two concepts are often confused. Authentication answers "who are you?" — it verifies your identity. Authorization answers "what are you allowed to do?" — it checks your permissions. IDOR is always an authorization failure, not an authentication one.

What IDOR looks like in the real world

IDOR is not limited to user profile endpoints. It shows up anywhere an application uses a predictable identifier to reference a resource. Some common patterns:

File downloads

Applications that store documents often reference them with sequential IDs:

GET /documents/download?id=892

Incrementing or decrementing that value may expose invoices, contracts, or medical records belonging to other users. The file is served based purely on the ID — no ownership check in sight.

API endpoints

Modern APIs are a goldmine for IDOR. Every endpoint that accepts a resource identifier is a potential target:

# These are all common IDOR targets GET /api/orders/5001 GET /api/messages/thread/88 PUT /api/invoices/3312/status DELETE /api/accounts/991

The risk is compounded by the fact that API responses are often more data-rich than web UI responses — a single IDOR on an API endpoint can leak significantly more sensitive information than the same flaw on an HTML page.

Multi-tenant SaaS platforms

Platforms that serve multiple organisations from a shared infrastructure are particularly vulnerable. A tenant isolation failure — where one customer can access another's data by manipulating a resource ID — is one of the most severe forms of IDOR and one of the hardest to catch in testing.

Why is this still happening in 2026?

This is the question worth dwelling on. The vulnerability class is well documented. The fix is conceptually simple. So why does it keep appearing?

The API explosion

A decade ago, a web application might have had dozens of server-side routes. Today, a mid-sized SaaS product might expose hundreds of API endpoints across REST, GraphQL, and internal service-to-service APIs. Each one of those endpoints is a potential authorization check that someone needs to write and maintain. The sheer volume makes gaps inevitable.

Microservices and inconsistent authorization

In a monolithic application, you might have one centralized middleware handling authorization logic. In a microservices architecture, each service often implements its own access control. When teams are moving fast and services are owned by different people, authorization policies become inconsistent. One service checks ownership. Another assumes the API gateway already handled it. Neither is right.

AI-generated code

This is the most interesting new wrinkle. AI coding assistants are genuinely good at generating working functionality quickly. Ask an LLM to write an endpoint that retrieves an order by ID and it will produce clean, functional code. What it often will not produce — unless you explicitly ask — is the authorization check that ensures the requesting user owns that order. The code works. The security layer is missing.

Authorization is hard to test automatically

Unlike SQL injection or XSS, IDOR requires business context to detect reliably. A scanner can identify that an endpoint accepts a numeric ID parameter. It cannot easily determine whether user A should be allowed to access resource B without understanding the application's data model. This means IDOR often slips through automated security scans and only surfaces during manual testing or in production via a bug bounty report.

How to find IDOR during testing

Manual testing is still the most reliable approach. The process is straightforward:

  1. Create two test accounts — call them User A and User B.
  2. As User A, perform actions that create or access resources: place an order, upload a file, send a message.
  3. Note all the identifiers that appear in requests — in URLs, request bodies, and headers.
  4. Switch to User B's session and attempt to access User A's resources using those identifiers.
  5. If User B can access User A's data, the application is vulnerable.

Burp Suite makes this significantly easier. You can use the Authorize extension to automatically replay requests with a different session token, flagging any responses that return the same data as the original request. It is not foolproof, but it catches a large percentage of simple IDOR issues quickly.

Tip from practice Do not just test GET requests. Some of the most impactful IDOR vulnerabilities are on write operations — PUT, PATCH, DELETE. Being able to delete another user's account or modify their subscription plan is significantly more damaging than just reading their profile.

What secure authorization looks like

The fix for IDOR is conceptually simple: never trust client-supplied identifiers without verifying ownership server-side. In practice, this means every data access query should include the authenticated user's identity as a constraint:

# Vulnerable — fetches the order regardless of who owns it order = Order.find(params[:id]) # Secure — scopes the query to the current user's orders only order = current_user.orders.find(params[:id])

The second version is not just more secure — it will raise a "record not found" error if the order belongs to a different user, so the behavior is also correct from a functional standpoint.

Use unpredictable identifiers

Replacing sequential integer IDs with UUIDs or random tokens does not fix the underlying authorization flaw, but it does make exploitation significantly harder. Guessing f8c1f3d9-b82f-4af9-a3c1-9d2e7b6f1a3c is a different challenge to incrementing 1042 to 1043. Use both — unpredictable IDs and proper authorization checks — not one as a substitute for the other.

Centralize authorization logic

In larger applications, duplicating authorization checks across every controller or handler is a recipe for gaps. A centralized authorization layer — whether that is a library, a middleware, or a policy framework — ensures consistent enforcement and makes the checks easier to audit.

Takeaway

IDOR is not a sophisticated vulnerability. It does not require exploit chains or advanced techniques. It requires one thing: that someone forgot to ask whether a user was allowed to access a resource. In 2026, with more API endpoints, more services, and more AI-generated code than ever, there are more places for that question to go unasked.

If you are learning web security, IDOR is one of the highest-value vulnerability classes to understand — both because it is so common and because finding it teaches you to think carefully about the gap between authentication and authorization, which underpins a huge proportion of real-world security issues.

Start with two accounts. Start changing IDs. See what the application tells you.
