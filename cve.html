CVE-2024-XXXXX Medium JWT misconfiguration
Algorithm confusion in [Vendor] authentication library allows token forgery
An open-source authentication library failed to enforce a server-side expected algorithm when verifying JSON Web Tokens, instead trusting the algorithm specified in the token header. This allowed an attacker in possession of the server's RSA public key to forge arbitrary tokens signed with HMAC-SHA256, effectively bypassing authentication for any application using the affected library version.

How I disclose

Responsible disclosure policy

My approach

When I discover a vulnerability — whether during authorised testing, personal research, or CTF-adjacent exploration — I follow a consistent responsible disclosure process. The goal is to give vendors enough time to patch before any public details are shared, while maintaining accountability for fixes actually happening.

Timeline

  • Initial report sent to vendor security contact within 24 hours of discovery
  • 90-day disclosure window from date of vendor acknowledgement
  • Extension granted on request if a patch is actively in progress
  • Public disclosure after patch release or at 90-day deadline, whichever comes first

What I include in reports

  • Clear description of the vulnerability and affected component
  • Step-by-step reproduction instructions
  • Proof-of-concept code or screenshots where relevant
  • CVSS score and my assessment of impact
  • Suggested remediation approach

Contact

Security researchers or vendors wanting to get in touch about a disclosure can reach me via the contact page. PGP key available on request.
