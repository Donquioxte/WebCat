Bypassing 2FA With a Race Condition — HackTheBox: Vault

Two-factor authentication is supposed to stop attackers in their tracks. On this box, a timing flaw in the OTP validation logic made it possible to bypass 2FA entirely using a multithreaded brute-force approach.

Published April 30, 2026 Reading time 14 min Author Wil

When "secure" authentication has a timing problem

Two-factor authentication is one of those controls that gets treated as a solved problem. Add a six-digit OTP step after the password, and the login flow is considered secure by default. This box was a good reminder that the implementation of 2FA matters just as much as the presence of it — and that a control which is logically sound but poorly implemented can be bypassed without ever touching the "secure" part directly.

The vulnerability here was not in the OTP generation itself. The code was a standard six-digit, time-based code. The flaw was in how the server validated attempts against it.

Mapping the login flow

The application's login process was a standard two-step flow: submit a username and password, then submit a six-digit OTP code on a second screen. Intercepting the requests in Burp Suite showed the structure clearly:

POST /api/auth/login {"username": "admin", "password": "********"} # Returns a temporary session token tied to the pending 2FA challenge {"status": "2fa_required", "challenge_token": "9f8a3c..."} POST /api/auth/verify-otp {"challenge_token": "9f8a3c...", "otp": "000000"}

The natural first question with any OTP verification endpoint is whether it rate-limits failed attempts. Submitting a handful of incorrect codes in sequence confirmed it did — after five failed attempts, the challenge token was invalidated and the user had to restart the login flow from the password step.

Five attempts against a six-digit code (one million possibilities) is nowhere near enough for a naive sequential brute-force. Rate limiting alone made this approach a dead end — at least, sequentially.

Finding the race condition

The rate-limiting logic worked by incrementing an attempt counter in the database after each failed OTP check, then checking that counter on the next request. The important detail — confirmed by testing — was that the counter increment happened after the OTP comparison completed, not before it.

Why this matters If the attempt counter is only checked and incremented sequentially, but several requests arrive at the server at nearly the same instant, each one can read the counter's value before any of the others have finished writing their increment. All of them see "4 attempts used" and proceed, even though five, ten, or twenty requests are in flight simultaneously.

This is a classic time-of-check to time-of-use (TOCTOU) race condition. The fix the developers intended — locking the account after five failed attempts — only works if attempts are processed one at a time. Fire enough requests at the server concurrently, and the rate limit's enforcement point can be raced past entirely.

Building the exploit

The approach: rather than guessing OTPs one at a time and respecting the rate limit, fire a large batch of guesses at the verification endpoint simultaneously, betting that several of them will land inside the same race window before the account locks.

Six-digit codes are still a million possibilities — too many to brute-force outright even with concurrency. But this server also revealed, through a slightly different bug, that the verification endpoint's response timing differed subtly when the first few digits of the OTP were correct versus completely wrong. That timing variation made a digit-by-digit approach far more practical than a flat brute-force of the full code space.

# Simplified concurrent request script (Python, using threads) import requests import threading URL = "https://target.htb/api/auth/verify-otp" TOKEN = "9f8a3c..." def try_otp(code): r = requests.post(URL, json={ "challenge_token": TOKEN, "otp": code }) if r.status_code == 200: print(f"[+] Valid OTP found: {code}") # Fire a batch of candidate codes concurrently to race the rate limiter candidates = [f"{i:06d}" for i in range(0, 50)] threads = [threading.Thread(target=try_otp, args=(c,)) for c in candidates] for t in threads: t.start() for t in threads: t.join()

Sending these requests using threads rather than sequentially meant they hit the server close enough together that several were processed before the rate-limit counter caught up. Repeating this in narrowed batches — using the timing signal to prioritise which digit ranges to test first — eventually surfaced the correct OTP without ever triggering a full lockout.

Tip from practice Burp Suite's Turbo Intruder extension is purpose-built for this kind of race condition testing — it can fire requests with much tighter timing precision than a basic threaded script, which matters a lot when the race window is only a few milliseconds wide.

Why this kind of bug is easy to miss

Race conditions in authentication flows are dangerous specifically because they are invisible in normal testing. A manual tester submitting OTPs one at a time, the way a real user would, will hit the rate limit exactly as intended and conclude the control works. The vulnerability only appears under concurrent load, which most functional and even most security testing does not naturally generate.

This is also a good example of why automated scanners rarely catch this class of bug — there is nothing syntactically wrong with the code. The logic is correct in isolation. It only breaks under a specific concurrency condition that has to be deliberately constructed.

How this gets fixed

  • Atomic counter operations. The attempt count and the lockout check need to happen as a single atomic database operation, not a separate read-then-write, so concurrent requests cannot all read the same stale value.
  • Application-level locking. A mutex or distributed lock keyed on the challenge token would force concurrent verification attempts for the same session to be processed sequentially rather than in parallel.
  • Consistent response timing. Verification responses should take a constant amount of time regardless of how much of the OTP was correct, removing the timing side-channel that made narrowing down the code practical in the first place.
  • IP and device-level throttling in addition to per-session counters, so that even a successful race against one challenge token doesn't scale into rapid-fire attempts across many sessions.

Takeaway

This box was a useful reminder that "we have rate limiting" is not the same as "we have rate limiting that holds up under concurrency." Race conditions are one of the more subtle vulnerability classes to test for, precisely because the obvious manual testing approach — trying things one at a time — can never reveal them. If you're testing authentication flows, it's worth deliberately asking: what happens if I send all of these requests at once instead of one after another?
