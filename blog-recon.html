My Go-To Recon Toolkit for Web App Engagements

Before a single payload gets fired, the outcome of an engagement is usually already being decided — by how thorough the recon was. Here's the toolkit I reach for first, and why each tool earns its place.

Published May 14, 2026 Reading time 10 min Author Wil

Why recon decides the engagement

It's tempting to think of penetration testing as the moment you find the clever injection point or chain together an exploit. In practice, most of the real work — and most of the value — happens before that, during recon. A target you only partially mapped is a target with vulnerabilities you'll never see, simply because you never knew that subdomain, that endpoint, or that forgotten staging environment existed.

This post walks through the recon workflow I use on web application engagements, the tools behind each step, and why I reach for them specifically.

Step 1: Subdomain enumeration

The attack surface of a modern web application is rarely just the main domain. Staging environments, internal admin panels, forgotten marketing microsites, and old API versions often live on subdomains that were never meant to stay exposed.

# Passive enumeration — pulls from certificate transparency logs, DNS aggregators etc. amass enum -passive -d target.com -o subdomains.txt # Active enumeration — brute forces against the actual DNS amass enum -active -d target.com -brute -o subdomains_active.txt

I run passive enumeration first since it generates no traffic to the target and casts a wide net using public data sources. Active enumeration follows once I have a baseline, since it catches subdomains that passive sources miss but generates DNS queries the target can see.

Don't skip certificate transparency A huge number of subdomains get discovered simply by checking crt.sh or similar certificate transparency logs. Any time an organisation issues a TLS certificate, the subdomain it covers becomes publicly logged — permanently. This is one of the highest signal, lowest effort recon steps available.

Step 2: Filtering for what's actually alive

A subdomain list of a few thousand entries is not useful until you know which ones are actually serving something. This is where httpx comes in — it probes each host and reports back status codes, titles, and technologies in use.

cat subdomains.txt | httpx -status-code -title -tech-detect -o alive_hosts.txt

This single command turns a flat list of names into a prioritised target list. A host returning a 200 with a login page and a visible CMS version is worth far more attention early on than one returning a generic 403.

Step 3: Content discovery

Once I know which hosts are alive, the next question is what's actually on them beyond the obvious pages. This is where directory and file brute-forcing earns its keep — looking for backup files, exposed config endpoints, admin panels, and old API routes that were never properly decommissioned.

# ffuf — fast, flexible, and good with custom wordlists ffuf -u https://target.com/FUZZ -w wordlist.txt -mc 200,301,302,403 -o ffuf_results.json

I tend to run this with a few different wordlists depending on the target's stack — a generic content discovery list first, then a more targeted one based on whatever framework httpx identified. A Laravel app and a Django app have very different files worth probing for.

API endpoint discovery

If the target exposes a JavaScript-heavy front end, the compiled JS bundles themselves are often a goldmine. Endpoint paths, internal API routes, and sometimes even hardcoded tokens get shipped to the client without anyone noticing.

# Pull all JS files, then grep for likely endpoint paths katana -u https://target.com -jc -o js_files.txt cat js_files.txt | grep -Eo '"\/[a-zA-Z0-9\/_-]+"' | sort -u

Step 4: Lightweight vulnerability scanning

Once I have a solid picture of the live attack surface, I run a fast, low-noise scan to catch known misconfigurations and common vulnerability patterns before diving into manual testing. This isn't a replacement for manual work — it's a way to avoid spending an hour manually testing for something a template-based scanner would have flagged in thirty seconds.

nuclei -l alive_hosts.txt -t cves/ -t misconfiguration/ -severity medium,high,critical -o nuclei_results.txt
Scanners find what they're told to look for Nuclei is only as good as its template set, and it will never replace understanding the application's business logic. Treat it as a triage step, not a substitute for manually walking through the app and thinking about how each feature could be abused.

Step 5: Keeping it organised

Recon generates a lot of output fast, and disorganised recon is nearly as bad as no recon at all if you can't act on what you found. I keep a simple folder structure per engagement:

target-name/ ├── recon/ │ ├── subdomains.txt │ ├── alive_hosts.txt │ ├── ffuf_results.json │ └── nuclei_results.txt ├── notes.md └── screenshots/

A plain markdown notes file might sound low-tech, but it forces me to write down hypotheses as I go — "this endpoint looks like it might be missing an ownership check, test later" — which has saved me from forgetting promising leads more than once.

Takeaway

None of these tools are exotic or hard to find — most appear on every "top recon tools" list you'll come across. The value isn't in knowing they exist, it's in running them consistently, in a sensible order, and actually reading the output carefully instead of rushing to the next tool. The widest possible view of the target, built methodically, is what makes the rest of the engagement easier.
