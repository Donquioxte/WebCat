Breaking JWT Authentication — Common Misconfigurations Developers Make

JSON Web Tokens are everywhere, and the spec itself is reasonably solid. The vulnerabilities almost always come from how it gets implemented — not from the standard itself.

Published March 10, 2026 Reading time 11 min Author Wil

A solid spec, undermined by implementation

JSON Web Tokens have become the default way modern applications handle stateless authentication. The format itself — a signed, base64-encoded JSON structure — is well-designed for its purpose. The vulnerabilities that show up in real-world testing almost never come from a flaw in the JWT specification. They come from developers trusting the token's contents without properly verifying how it was signed, or implementing the verification logic in a way that quietly accepts tokens it shouldn't.

This post walks through the JWT misconfigurations I encounter most often during web application assessments, starting with the structure of a token itself.

The anatomy of a JWT

A JWT consists of three base64url-encoded segments separated by dots: a header, a payload, and a signature.

# Header — declares the signing algorithm {"alg": "HS256", "typ": "JWT"} # Payload — the actual claims (user data, expiry, etc.) {"sub": "1042", "role": "user", "exp": 1781234567} # All three segments combined and signed header.payload.signature

The critical detail is that the header and payload are only encoded, not encrypted. Anyone can decode and read them without knowing the signing key. The signature is the only thing standing between "this token says I'm an admin" and the server actually believing it. Every vulnerability in this post is really a variation on attackers finding a way to make the server accept a forged or manipulated signature — or skip verifying it at all.

The "none" algorithm attack

The JWT specification includes a legitimate none algorithm, intended for cases where the token doesn't need a signature at all. Some JWT libraries, particularly older or poorly maintained ones, will honour this header value and skip signature verification entirely if the server's verification code doesn't explicitly reject it.

# Original token header {"alg": "HS256", "typ": "JWT"} # Attacker-modified header, with the signature stripped {"alg": "none", "typ": "JWT"}

If the verification library trusts the algorithm specified in the token's own header rather than enforcing an expected algorithm server-side, an attacker can simply set alg to none, modify the payload to whatever they want — "role": "admin", for instance — and remove the signature segment entirely. A vulnerable server will accept this token as valid.

Why this still happens The root cause is almost always the same: the server's verification logic reads the algorithm to use from the token itself, instead of having a hardcoded, expected algorithm configured independently. Trusting client-supplied input to decide how that same input should be verified is the recurring theme across nearly every JWT vulnerability.

Algorithm confusion — RS256 to HS256

This is a more subtle and more damaging variant. Many applications use RS256, an asymmetric algorithm where the server signs tokens with a private key and verifies them with a corresponding public key. The public key, by design, is not secret — it is often even exposed at a known endpoint or embedded in client-side code.

The vulnerability occurs when the server's verification code does not distinguish between algorithms correctly. Some JWT libraries pass whatever key is configured into a generic "verify" function based on the algorithm specified in the token header — and if an attacker sets the header to HS256 (a symmetric algorithm using a shared secret), the library may use the public RSA key as the HMAC secret for verification:

# Forging a token using the server's own public key as the HMAC secret import jwt public_key = open("server_public_key.pem").read() forged_token = jwt.encode( {"sub": "1042", "role": "admin"}, public_key, algorithm="HS256" )

Since the attacker knows the public key (it's public, after all) and the server is unknowingly using it as an HMAC secret to verify HS256 signatures, the forged token passes verification. This single misconfiguration can fully compromise the token-based trust model — effectively letting an attacker sign their own admin tokens.

Weak or guessable HMAC secrets

For applications using HS256 correctly, the security of every issued token depends entirely on the secrecy and strength of the signing key. Short, predictable, or accidentally committed secrets are common enough that brute-forcing them is a standard part of JWT testing.

# Brute-forcing a weak HMAC secret offline using a wordlist jwt_tool eyJhbGciOiJIUzI1NiJ9... -C -d /usr/share/wordlists/rockyou.txt

Once the secret is recovered, an attacker can sign arbitrary tokens with any claims they like — full account takeover, privilege escalation, the works — entirely offline, without ever interacting with the live server again until they submit the forged token.

Missing expiry and claim validation

Even with a properly signed token, the application still needs to validate its claims on every request — not just at issuance. Common gaps include:

  • Not checking exp server-side, relying instead on the client to discard expired tokens, which an attacker can simply ignore.
  • Not validating the aud (audience) claim, allowing a token issued for one service to be replayed against a different one that shares the same signing key.
  • Trusting role or permission claims embedded in the token without re-checking them against the current state of the user's account — meaning a role change or account suspension has no effect until the token naturally expires.

How I test for JWT issues

  1. Decode the token and inspect the header and payload for anything unexpected — role claims, internal IDs, debug flags.
  2. Try the none algorithm attack by stripping the signature and setting alg to none.
  3. If RS256 is in use, attempt the algorithm confusion attack using any discoverable public key.
  4. Attempt to brute-force the signing secret offline if HS256 is used.
  5. Tamper with claims (role, user ID, expiry) and resubmit with the original signature to confirm the server actually re-validates the signature against the modified payload.
  6. Check whether an expired token is still accepted by the server.

Tools like jwt_tool automate most of this process and are a good starting point, but understanding what each test is actually checking for matters more than memorising the tool's flags.

How this gets fixed

  • Hardcode the expected algorithm on the server side rather than reading it from the token header.
  • Never use the same key type for both signing and verification across algorithm types — keep RS256 and HS256 verification logic strictly separate.
  • Use long, randomly generated secrets for HS256 — at least 256 bits of entropy, stored outside of source control.
  • Validate every claim that matters on every request — expiry, audience, issuer — not just at login.
  • Keep token lifetimes short and pair them with a refresh token flow, so a compromised access token has a limited window of usefulness.

Takeaway

JWT vulnerabilities are a good example of a broader pattern in security testing: the underlying technology is sound, but trusting client-controlled input to determine how that same input gets validated creates a loophole almost every time. When testing any JWT-based authentication system, the first question worth asking is not "is the signature valid" but "what is actually deciding how the signature gets checked" — and whether an attacker has any influence over that decision.
