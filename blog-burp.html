Burp Suite Extensions That Actually Save Time

The BApp Store has hundreds of extensions. Most aren't worth installing. These are the ones I actually keep enabled on every engagement — and why each one earns its place.

Published March 28, 2026 Reading time 9 min Author Wil

Curation beats collection

It's tempting, especially early on, to install every extension in the BApp Store that sounds useful. In practice, a cluttered extension list slows Burp down, adds noise to your scan results, and makes it harder to find the tool you actually need in the moment. A small, well-chosen set of extensions consistently outperforms a large unfocused one.

These are the extensions that have earned a permanent place in my setup, grouped by what stage of testing they help with.

Authorize — automated access control testing

Authorize is built specifically for testing broken access control and IDOR-style vulnerabilities. Once configured with a low-privilege session token, it automatically replays every request that passes through Burp using that lower-privilege session, and flags any response that returns the same data as the original higher-privilege request.

# Conceptually, Authorize is doing this for every request automatically: GET /api/orders/5001 [as User A — 200 OK, full order data] GET /api/orders/5001 [replayed as User B via Authorize — 200 OK?]

Running this passively in the background while you browse the application manually means access control gaps surface as you go, instead of requiring a separate dedicated pass at the end of testing. It won't catch every IDOR — business logic context still matters — but it catches a large share of the obvious ones with almost no extra effort.

JWT Editor — manipulating tokens in-flight

For any engagement involving JWT-based authentication, this extension is close to essential. It adds a dedicated tab for decoding, editing, and re-signing JSON Web Tokens directly inside Burp, without needing to shell out to a separate CLI tool every time you want to test a claim manipulation.

Why this beats manual decoding JWT Editor lets you generate and manage signing keys directly inside Burp, then test algorithm confusion and signature-stripping attacks by editing the token and immediately re-sending the request — all without leaving the proxy history. It turns JWT testing from a multi-tool workflow into a few clicks.

Param Miner — finding hidden parameters and headers

Param Miner brute-forces hidden, unlinked parameters and HTTP headers that an application's backend silently accepts but never exposes in its visible UI. This matters because hidden parameters are disproportionately likely to be missing proper validation — nobody is actively maintaining input handling for a parameter they forgot exists.

# Conceptually, Param Miner sends variations like: GET /api/search?q=test&debug=true GET /api/search?q=test&internal_override=1 GET /api/search?q=test X-Forwarded-Host: attacker-controlled.com

It's also one of the more reliable tools for surfacing web cache poisoning vulnerabilities, since it's specifically built to detect unkeyed inputs that influence cached responses — a class of bug that is genuinely hard to find by hand.

Turbo Intruder — high-precision timing attacks

Standard Burp Intruder is fine for most brute-force tasks, but it isn't built for attacks where request timing matters down to the millisecond — race conditions being the clearest example. Turbo Intruder uses a custom HTTP engine that can fire a large batch of requests with much tighter timing precision than the default tooling.

# Simplified Turbo Intruder script structure def queueRequests(target, wordlists): engine = RequestEngine(endpoint=target.endpoint, concurrentConnections=20, requestsPerConnection=10) for i in range(20): engine.queue(target.req, gate='race1') engine.openGate('race1')

This is the tool I reach for whenever a finding depends on getting multiple requests to land inside a narrow server-side time window — exactly the kind of scenario that comes up when testing rate limiting or account-locking logic for race conditions.

Logger++ — better history and filtering

Burp's default proxy history is functional but limited once a session has thousands of requests logged. Logger++ adds far more powerful filtering, search, and export options across all of Burp's tools, not just the proxy — including the ability to filter by response time, which is genuinely useful when hunting for the kind of timing anomalies that hint at SSRF or blind injection.

Tip from practice Filtering Logger++ by response time descending is a quick, low-effort way to spot endpoints that might be making slow backend calls — a common signal worth investigating further for SSRF, blind SQL injection, or inefficient query patterns that could indicate a denial-of-service angle.

A note on extension overlap

A few extensions in the BApp Store solve overlapping problems — there are multiple authorization-testing extensions, multiple JWT tools, and so on. Rather than installing several that do roughly the same job, I'd rather pick one per category, learn its quirks properly, and keep the overall extension list lean. A smaller, well-understood toolkit is faster in practice than a comprehensive one you have to relearn every engagement.

Takeaway

None of these extensions replace manual testing or careful thinking about an application's logic — they remove repetitive, mechanical work so more time goes toward the parts of testing that actually require a human. If you're building out your own Burp setup, I'd suggest starting with just Authorize and JWT Editor, since they cover two of the highest-value, most common vulnerability classes, and adding the others as you run into the specific problems they solve.
